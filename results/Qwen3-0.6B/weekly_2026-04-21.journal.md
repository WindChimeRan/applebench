## Fixes Applied

### mlx_lm (chat, full; agent, c=1 only) — local mlx pin
- **Error**: `RuntimeError: There is no Stream(gpu, 0) in current thread.` thrown from `mlx_lm/generate.py:1161` inside `BatchPromptProcessor.prompt` at `mx.eval([c.state for c in self.prompt_cache])`. Happens on the `Thread-1 (_generate)` worker thread that mlx_lm.server uses for batch generation. HTTP 200 returns with an empty SSE body → benchmark.py's silent-failure detector → Phase 1 hang.
- **Diagnosis**: mlx 0.31.2 made GPU streams thread-local. mlx_lm creates `generation_stream = mx.new_stream(mx.default_device())` at module import (main thread) but then uses it inside a worker thread via `with mx.stream(generation_stream):`. Under 0.31.2 that stream isn't visible to the worker thread, so `mx.eval` fails.
- **Reference**: mlx 0.31.1 vs 0.31.2 (stream-thread-local change). Upstream mlx_lm main had not caught up at the time of this run.
- **Fix**: first pinned `mlx==0.31.1 mlx-metal==0.31.1` in `install_mlx_lm.sh` and `update_mlx_lm.sh` (commit `4b7f78e`). After confirming the real upstream fix (`ed1fca4 Thread local generation stream` by Angelos Katharopoulos, 2026-04-22 00:34) exists on mlx-lm main, switched both scripts to track `mlx-lm @ git+https://github.com/ml-explore/mlx-lm` (commit `b7afa33`), dropping the mlx pin. The git path mirrors how `install_omlx.sh` already resolves mlx-lm. Revert to PyPI when a release contains `ed1fca4`.
- **Verification**: For both the pin and the git-main switch: raw `curl /v1/chat/completions` returned tokens, `benchmark.py --requests 3 --concurrency 1` passed, then full reruns on both splits with `--skip-existing mlx_lm APPLEBENCH_METALSTAT=1`.
- **Result (pin, 17194c0)**: chat 100/0 all 3 levels (138.6 / 177.9 / 185.5 tok/s). agent c=1 100/0 (86.4 tok/s), c=8/c=16 OOM.
- **Result (git-main, below)**: chat 100/0 all 3 levels (108.6 / 173.6 / 182.6 tok/s — cc=1 ~22% lower than pin, cc=8/16 within 2%; could be mlx 0.31.1→0.31.2 single-stream perf change or thermal from multiple rerun cycles, worth revisiting next week with fresh thermals). agent c=1 100/0 (73.3 tok/s), c=8 19/81, c=16 0/100 — same Metal OOM pattern as pin, confirming it's KV-pressure rather than mlx version.
- **Commits**: `4b7f78e` (pin), `17194c0` (pin results), `b7afa33` (git-main switch), results commit below.

### vllm_metal (chat + agent) — upstream fix
- **Error**: `error: no member named 'add_temporary' in 'mlx::core::metal::Device'` (and `get_command_encoder`) when JIT-compiling `metal/paged_ops.cpp`. Original Phase 1 run hit this on both splits, aborting `run_all.sh` via `set -e` on the serve script.

### mlx_lm (agent c=8/c=16) — Metal OOM after pin
- **Observation**: Once the mlx pin unblocked c=1 (100/0), c=8 dropped to 19/81 and c=16 to 0/100. Server log shows `libc++abi: terminating due to uncaught exception of type std::runtime_error: [METAL] Command buffer execution failed: Insufficient Memory (00000008:kIOGPUCommandBufferCallbackErrorOutOfMemory)` immediately after the Prompt Cache report showed "10 sequences, 7.37 GB". Last week c=8 was 100/0 — so this is a fresh symptom exposed by the pin rather than a pre-existing limit, but it's downstream of the same mlx-lm release: with 0.31.1+ prompt_cache and 4K-prompt agents, 8 concurrent sequences overflow the Metal heap.
- **Note**: Not auto-fixed. A possible mitigation is capping agent concurrency for mlx_lm or lowering `--max-kv-size`, but both are scope-extension (mlx_lm doesn't expose a concurrency cap in its serve script, and benchmark defaults should stay consistent across frameworks for comparability). Flag for triage.

### hf_transformers (agent c=8/c=16) — degraded
- **Observation**: c=8 74/26 (TTFT p50 113s), c=16 29/71 (TTFT p50 152s). First weekly run, so no week-over-week baseline.
- **Note**: Not auto-fixed. This is hf_transformers' first agent benchmark; degradation at high concurrency with 4K prompts is not unexpected for an MPS-backed reference implementation. Flagging so future weeks can establish whether this is steady state or a regression.
