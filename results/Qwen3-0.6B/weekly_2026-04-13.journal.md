### inferrs (chat + agent) — commit 61d87a9 (memory cap)
- **Error**: Not a failure — investigation of memory pressure triggered by user question about which framework "ate all memory." inferrs server log showed `Paged attention memory baseline: 51.84 GiB, fraction=0.90, Paged KV store: 27300 blocks × 16 tokens/block`. That's 90% of system RAM preallocated for KV, way more than Qwen3-0.6B needs, and an observed side-effect was that vllm_metal subsequently saw only 34.4 GB of Metal's 55 GB — the previous framework hadn't released.
- **Diagnosis**: inferrs's `--paged-attention` flag accepts an explicit fraction but our serve script passed it bare, silently defaulting to 0.9. The preallocation wasn't just wasteful — it turned out to cause **cache thrashing** that slowed inferrs c=1 from expected ~14 min/level to the ~100 min/level we'd been observing.
- **Reference**: inferrs CLI help: `--paged-attention[=<FRACTION>]`
- **Fix**: `scripts/serve_inferrs.sh`: `--paged-attention` → `--paged-attention=0.5`
- **Verification**: Startup log confirms `fraction=0.50, Paged KV store: 15166 blocks × 16 tokens/block = 242656 total slots` (was 27300 blocks). Subsequent benchmark c1 wall time: **847s** (was projected ~6000s at the old fraction). c8 wall: **753s**. Effectively 5–7× faster per level — and the chat retry completed in ~45 min instead of the 2 hours I was planning around.
- **Commit**: `61d87a9` (auto-fix(inferrs + vllm_mlx): cap Metal memory reservation at 50%)

## Reverts

### vllm_mlx (agent) — commit 490186d
The same `61d87a9` commit also added `--gpu-memory-utilization 0.5` to `serve_vllm_mlx.sh`. That change was reverted 20 minutes later after a failed retry showed it crashed at agent c=8 with `[METAL] Command buffer execution failed: Insufficient Memory (kIOGPUCommandBufferCallbackErrorOutOfMemory)`. The vllm_mlx server log made the mechanism clear:
```
[Memory pressure] 42.5GB > 38GB threshold, forced cache clear
... active=42.5GB peak=43.6GB cache=0.0GB step=26624 running=8
[Memory pressure] 49.9GB > 38GB threshold, forced cache clear
libc++abi: terminating due to uncaught exception of type std::runtime_error: [METAL] Command buffer execution failed: Insufficient Memory
```
The `--gpu-memory-utilization` flag is **advisory only** in vllm_mlx 0.2.8 — the force-cache-clear threshold is hardcoded at 38 GB regardless of the configured fraction, so the 0.5 cap (nominal ~28 GB limit) did not prevent allocations from climbing to 50 GB and OOMing Metal. Reverting the flag removes a variable without removing protection. Confirmed on the subsequent retry that vllm_mlx agent c=8 fails the **same way** with and without the cap — it's an upstream fragility, not a regression introduced by our fix. Kept the inferrs cap because inferrs does enforce its fraction strictly.

## Memory observations (bonus)

During this run, the user asked me to investigate which framework "ate all memory" and whether we could cap them from our benchmark config side. Findings worth preserving for next week's skill author:

- **inferrs** preallocates 90% of system RAM by default via `--paged-attention`. Capping to `0.5` is a one-flag fix in `serve_inferrs.sh` and — unexpectedly — made inferrs **5–6× faster** because the oversized preallocation was causing cache thrashing. Applied.
- **vllm_mlx** also defaults to ~90% (`allocation_limit=50.1GB (90% of 55.7GB)`) but its `--gpu-memory-utilization` flag is advisory only; the internal force-cache-clear threshold is hardcoded at 38 GB and Metal lets it allocate past that anyway. Capping from our side does not prevent the Metal OOM at c=8+ with 4K prompts. Not applied — same failure either way.
- **ollama** (observed 54 GB RSS during its agent run) has no CLI cap. Only env vars: `OLLAMA_GPU_OVERHEAD` (reserve N bytes away from ollama), `OLLAMA_CONTEXT_LENGTH`, `OLLAMA_NUM_PARALLEL`, `OLLAMA_KV_CACHE_TYPE=q8_0` (quantize to halve KV). None is a direct memory cap. Not applied this run — deferred pending user confirmation on which env vars are acceptable (q8_0 would change ollama's output parity vs the bf16 baseline all other frameworks use).
